Late-Interaction on turbopuffer

Late interaction is worth shipping on turbopuffer today, even before native engine support lands. The practical takeaway from this project is that Pattern C, Dense ANN first stage plus ColBERT MaxSim rerank, is the default recipe to recommend: on this benchmark it beats Dense by 46% R@10 on Amazon (0.198 vs 0.135), by 3.9% R@10 on SciFact (0.807 vs 0.777), and ties Dense on Quora (0.814 vs 0.813) while staying at the same query cost as Dense and requiring no extra turbopuffer storage.

What late interaction is

Dense retrieval stores one vector per document and compares one query vector to one document vector. It is fast and robust, but a pooled vector can blur together multiple concepts or rare terms.

Cross-encoder reranking lets the model compare the whole query and the whole document jointly, which is usually stronger, but it is too expensive to use as a first-stage retriever.

Late interaction is the middle ground. Documents are encoded offline into many vectors, usually one per token. Queries are also encoded into many vectors. At search time, the score is computed by letting each query token find its best-matching document token and then summing those best matches.

                  M
S(Q, D) =         ОЈ   max  q_i В· d_j
                 i=1   j

That scoring function is MaxSim. It is the core primitive behind ColBERT-style retrieval.

Why it matters on turbopuffer today

turbopuffer does not yet ship a native multi-vector column type or a server-side MaxSim operator. That is the product gap. The customer need already exists, and the current API surface is still enough to ship a credible answer.

  • ANN vector retrieval gives a strong first-stage candidate set.
  • multi_query supports the token fan-out needed for Pattern B.
  • Local reranking lets the client compute exact or approximate MaxSim today.
  • The dense namespace stays cheap enough that Pattern C keeps Dense economics.
The practical question is not whether native support would be cleaner. It would. The practical question is what to recommend to customers before that lands.

Three patterns on turbopuffer today

Pattern Query path Why you would use it Main tradeoff
Pattern A - ColBERT rerank pooled-ColBERT ANN first stage, local MaxSim rerank Single-encoder setup; useful fallback on long-doc workloads Candidate-set quality is limited by pooled-ColBERT
Pattern B - token-as-row one ANN lookup per query token through multi_query, then client aggregation When token matrices cannot live on local disk Higher latency, two query calls, row-count blow-up
Pattern C - Dense + MaxSim rerank existing dense ANN first stage, local MaxSim rerank Recommended default; strongest first-stage retrieval plus token-level precision Requires dense and ColBERT encoders at query time

Why Pattern C is the headline. It composes with an index customers often already have. There is no re-ingest of a new turbopuffer namespace, no extra turbopuffer storage, and the query still uses one ANN call before reranking locally.

Benchmark evidence

The benchmark covers three intentionally different retrieval profiles:

Dataset Why it matters Corpus / queries
Quora Duplicates Paraphrase-heavy, dense-friendly anchor 149,596 docs / 5,000 queries
Amazon Reviews 2023 Vocabulary-leaning retrieval with long-tail product language 25,000 docs / 2,000 queries
BEIR/SciFact Scientific claim to abstract retrieval with strong length asymmetry 5,183 docs / 300 queries

Amazon: Pattern C is the breakout winner

Pipeline Recall@10 MRR@10 nDCG@10 p50 latency (ms) $ / M searches
Dense baseline (MiniLM) 0.135 0.083 0.096 60.4 4.00
Pattern A - ColBERT rerank 0.046 0.039 0.040 62.1 4.00
Pattern B - token-as-row 0.140 0.097 0.107 135.4 8.00
Pattern C - Dense + MaxSim rerank 0.198 0.149 0.161 65.2 4.00
  • Headline: Pattern C beats Dense by +46% R@10, +79% MRR@10, and +68% nDCG@10.
  • Confidence: paired-delta confidence intervals exclude zero on all three quality metrics.
  • Interpretation: Pattern A fails here because pooled-ColBERT is the weak link. Pattern C fixes the first stage by inheriting MiniLM recall and then using MaxSim for precision.
  • Important nuance: Pattern B also beats Dense on Amazon and reaches 0.503 R@10 vs 0.389 Dense on 10+ token queries. Pattern C is still the stronger operator recommendation.

Cost, latency, and storage tradeoffs

The operator decision table from results/cost-analysis.md compresses to this:

Mode Query cost Latency profile Storage profile Practical reading
Dense baseline $4.00 / M Lowest Lowest Keep this on short-query, dense-friendly workloads
Pattern A $4.00 / M Moderate Low in turbopuffer, local token store required Use when only one encoder is feasible
Pattern B $8.00 / M Highest 30-100x row count, best with f16 Use when local token matrices are the bottleneck
Pattern C $4.00 / M Near Dense No extra turbopuffer storage Default recommendation

Recommended guidance

  1. Default: recommend Pattern C first when the customer can run a dense encoder and a ColBERT encoder.
  2. If token matrices cannot live locally: recommend Pattern B, ideally with f16.
  3. If only one encoder is feasible at query time: recommend Pattern A with a larger candidate set, especially for long-document retrieval.
  4. If the workload is short-query paraphrase search: keep Dense as the primary retriever and consider fusion with Pattern C for recall headroom.

Live demo plan

The live demo should prove mechanics, not claim quality. The safest command is:

make demo Q="why is the sky blue"

That query is stable against the Quora token namespace and shows the full Pattern B flow clearly:

  1. encode the query with jinaai/jina-colbert-v2
  2. fan out one ANN lookup per query token through multi_query
  3. aggregate by doc_id
  4. rank by approximated MaxSim

The exact presenter script lives in DEMO.md. The reason to keep the live demo on Pattern B while recommending Pattern C is simple: Pattern B is the clearest mechanics demo; Pattern C is the strongest production result.

Roadmap

The internal roadmap in guide/ROADMAP.md argues for a three-step ladder:

Priority Proposal Why it matters
P0 Multi-vector attribute type: [K,D]f16 / [K,D]f32 Eliminates Pattern B's row-count explosion and Pattern A/C's external token-matrix store
P1.5 MUVERA-style single-vector summary Fastest path to a customer-facing late-interaction surface using the existing ANN machinery
P1 Native exact MaxSim rank_by operator Collapses client fan-out and local rerank into one server-side path
  • P0 first removes the current storage-shape workaround.
  • P1.5 next lands a credible customer-facing story quickly by reusing the existing ANN machinery.
  • P1 after that becomes the exact long-term answer and composes naturally with MUVERA as a rerank stage.
